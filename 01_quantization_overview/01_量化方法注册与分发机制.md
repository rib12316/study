# 特性 #1：vLLM 量化方法注册与分发机制（Quantization Method Registry & Dispatch）

> 学习阶段：AI Infra 基础储备 / 量化方向入门
> 对应源码版本：vllm-project/vllm @ main（2026-06 克隆）
> 本讲定位：所有具体量化方法（AWQ / GPTQ / FP8 …）的**共同骨架**。学完它，后续每一讲学具体方法时只需关注"数学 + kernel"。

---

## 一、这一讲为什么重要（给你这个量化方向的研究生）

你熟悉量化的**数学**（scale / zero-point / per-channel / 对称非对称 / 校准）。但把数学变成一个能在推理引擎里跑起来的特性，需要解决四个工程问题：

1. **识别**：用户丢过来一个 HF checkpoint，引擎怎么知道它是 AWQ 还是 GPTQ？
2. **选择 kernel**：同一个数学量化（比如 W4A16），不同硬件（A100 / H100 / 老 V100）要用不同的 CUDA kernel（GPTQ-Marlin / Machete / cutlass），怎么选？
3. **替换权重布局**：量化的权重在显存里的排布（pack、interleave、scale 存放）和普通 Linear 完全不同，怎么让 nn.Linear 用量化权重？
4. **可扩展**：明天有人发明了新量化方法，怎么让社区一行代码接进来而不用改 vLLM 核心？

vLLM 用一套 **"注册表 + 工厂 + 策略"** 的设计模式优雅地解决了这四点。这是大型推理框架里非常典型的架构，理解它对你做 AI Infra 工作（尤其面试）是基本功。

## 四、注册表与工厂（步骤 ③）

### 4.1 `QuantizationMethods`：菜单

`vllm/model_executor/layers/quantization/__init__.py:12`

```python
QuantizationMethods = Literal[
    "awq", "auto_awq", "fp8", "fbgemm_fp8", "fp_quant",
    "modelopt", "modelopt_fp4", "modelopt_mxfp8", "modelopt_mixed",
    "auto_gptq", "gptq", "gptq_marlin", "awq_marlin",
    "humming", "compressed-tensors", "bitsandbytes", "experts_int8",
    "quark", "moe_wna16", "torchao", "inc", "mxfp4", "gpt_oss_mxfp4",
    "deepseek_v4_fp8", "online",
    "fp8_per_tensor", "fp8_per_block", "fp8_per_channel",
    "int8_per_channel_weight_only", "mxfp8",
]
QUANTIZATION_METHODS: list[str] = list(get_args(QuantizationMethods))
```

**设计要点**：这是一个 `Literal` 类型，本质是个"白名单"。它同时是 IDE 提示、运行时校验、文档三合一。`QUANTIZATION_METHODS` 列表则用于运行时检查 `if name in QUANTIZATION_METHODS`。

> 💡 面试点：为什么用 `Literal` 而不是 `Enum`？—— `Literal` 在静态类型检查（mypy/pyright）下能把字符串值传递下去做 narrowing，而 Enum 在和 HF config 的 dict 字符串交互时需要反复转换，对这种"字符串驱动"的系统不友好。

### 4.2 `get_quantization_config`：工厂（核心！）

`vllm/model_executor/layers/quantization/__init__.py:108`

```python
def get_quantization_config(quantization: str) -> type[QuantizationConfig]:
    if quantization not in QUANTIZATION_METHODS:
        raise ValueError(f"Invalid quantization method: {quantization}")

    # lazy import —— 故意把 import 放在函数体内
    from .auto_awq import AutoAWQConfig
    from .auto_gptq import AutoGPTQConfig
    from .fp8 import Fp8Config
    # ... 省略一堆 import

    method_to_config: dict[str, type[QuantizationConfig]] = {
        "awq": AutoAWQConfig,
        "awq_marlin": AutoAWQConfig,   # ← 注意：多个名字指向同一个类
        "gptq": AutoGPTQConfig,
        "gptq_marlin": AutoGPTQConfig, # ← 同一个类内部再决定用不用 Marlin
        "fp8": Fp8Config,
        "compressed-tensors": CompressedTensorsConfig,
        # ...
    }
    return method_to_config[quantization]
```

**三个设计精华你要记住**：

1. **Lazy import（延迟导入）**：所有 `from .fp8 import Fp8Config` 都写在函数体里，而不是文件顶部。原因是某些量化后端（如 `torchao`、`mxfp4`）会触发 `torch.compile` 或导入 Triton，在模块加载阶段就执行会拖慢启动甚至报错。延迟到真正需要时才导入，是大型项目的常见优化。

2. **多对一映射**：`"awq"`、`"awq_marlin"`、`"auto_awq"` 三个名字都映射到 `AutoAWQConfig`。"用不用 Marlin"这个决策**不在工厂层做**，而是下沉到 `AutoAWQConfig.get_quant_method()` 里根据硬件能力决定（见第六节）。这是"工厂只负责选策略类，策略类内部再选实现"的分层。

3. **开放扩展点**：函数最后有
   ```python
   method_to_config.update(_CUSTOMIZED_METHOD_TO_QUANT_CONFIG)
   ```
   外部插件可以通过 `@register_quantization_config("my_quant")` 装饰器往这个 dict 注入新方法，不用改 vLLM 源码。这就是问题④（可扩展）的答案。

### 4.3 自定义注册装饰器

`__init__.py:58`

```python
_CUSTOMIZED_METHOD_TO_QUANT_CONFIG = {}

def register_quantization_config(quantization: str):
    def _wrapper(quant_config_cls):
        if quantization in QUANTIZATION_METHODS:
            logger.debug(...)        # 已存在则覆盖
        else:
            QUANTIZATION_METHODS.append(quantization)
            if sq := current_platform.supported_quantization:
                sq.append(quantization)   # 同步注册到平台支持列表
        if not issubclass(quant_config_cls, QuantizationConfig):
            raise ValueError(...)
        _CUSTOMIZED_METHOD_TO_QUANT_CONFIG[quantization] = quant_config_cls
        return quant_config_cls
    return _wrapper
```

用法：
```python
@register_quantization_config("my_quant")
class MyQuantConfig(QuantizationConfig):
    ...
```

> 💡 这是 vLLM 整个生态的通用模式：核心子系统（attention backend、platform、quant method、loader）都提供 `register_xxx` 装饰器，让 out-of-tree 扩展成为一等公民。你以后看 attention backend 注册时会看到一模一样的套路。

## 五、两个抽象基类（策略与实现的契约）

`vllm/model_executor/layers/quantization/base_config.py`

### 5.1 `QuantizationConfig`（策略 = 菜谱）

每种量化方法都要实现这个抽象类。它的核心抽象方法：

| 方法 | 作用 | 何时调用 |
|------|------|----------|
| `get_name()` | 返回量化方法名（如 "fp8"） | 调试/日志 |
| `get_supported_act_dtypes()` | 支持哪些激活 dtype | 校验（vllm.py:630） |
| `get_min_capability()` | 最低 GPU 算力（70=Volta, 80=Ampere, 90=Hopper） | 硬件校验（vllm.py:623） |
| `get_config_filenames()` | 去模型目录找哪些 json 文件读配置 | 旧式 checkpoint |
| `from_config(cfg: dict)` | 从 HF config dict 构造本类实例 | 步骤④ |
| `get_quant_method(layer, prefix)` | **为某一层返回具体的 QuantizeMethodBase** | 步骤⑤（最关键！） |
| `override_quantization_method(...)` | "我这个类能不能接管这个 checkpoint？" | 见第六节 |

还有两个非抽象但重要的字段/方法：
- `self.packed_modules_mapping: dict[str, list[str]]`：把模型里融合的权重（如 `qkv_proj`）映射回三个独立权重（`q_proj/k_proj/v_proj`），用于量化 scale 对齐。
- `get_cache_scale_mapper()`：KV cache 量化的 scale 名字重映射。

### 5.2 `QuantizeMethodBase`（实现 = 厨师）

每一层（每个 Linear）持有一个 `quant_method` 对象。核心抽象方法只有两个必须实现：

```python
class QuantizeMethodBase(ABC):
    uses_meta_device: bool = False   # 在线量化时用 meta device 省峰值显存

    @abstractmethod
    def create_weights(self, layer, *weight_args, **extra_weight_attrs):
        """在 layer 上创建量化所需的参数（weight, scale, ...）"""
        ...

    @abstractmethod
    def apply(self, layer, *args, **kwargs) -> torch.Tensor:
        """前向计算：用量化权重 × 输入"""
        ...

    def process_weights_after_loading(self, layer) -> None:
        """权重全部加载完后做后处理（转置/重排/在线量化等）。默认空实现"""
        return
```

**这三步生命周期是你以后读任何量化方法实现都要找的三件事**：
1. `create_weights` —— 这层要存哪些张量？什么 shape/dtype？
2. `process_weights_after_loading` —— 权重从 checkpoint 进来后要怎么变换才能给 kernel 用？
3. `apply` —— 实际调哪个 kernel？

把这三步记成 **C-P-A**（Create / Process / Apply），后面每一讲的具体量化方法你都用这个框架去拆。

## 六、override 机制：为什么同一个 quant_method 名字会路由到不同的实现？（步骤 ②③ 的隐藏层）

这是 vLLM 量化系统**最巧妙、也最容易被忽略**的设计，必须单独讲。

### 6.1 问题

HF checkpoint 里写 `"quant_method": "gptq"`。但 vLLM 内部有 `gptq`、`gptq_marlin`、`auto_gptq` 三个名字。到底用哪个？而且：用户可能显式传 `--quantization gptq_marlin` 想强制走 Marlin kernel。

### 6.2 解决：`_verify_quantization` 里的 override 优先级探测

`vllm/config/model.py:970`

```python
def _verify_quantization(self) -> None:
    quant_cfg = self.model_arch_config.quantization_config   # HF 里的 dict
    if quant_cfg is not None:
        quant_method = quant_cfg["quant_method"]             # "gptq"

        # 按优先级排好的"候选接管者"清单
        overrides = ["auto_gptq", "gptq", "gptq_marlin",
                     "auto_awq", "awq", "awq_marlin", "inc", ...]

        # 遍历每一个候选，问它："你能接管这个 checkpoint 吗？"
        for name in overrides:
            method = me_quant.get_quantization_config(name)
            quantization_override = method.override_quantization_method(
                quant_cfg, self.quantization, hf_config=self.hf_config
            )
            if quantization_override is not None:
                self.quantization = quantization_override   # 接管！
                break
```

每个候选 Config 类实现 `override_quantization_method`，返回一个非 None 值表示"我要接管"。看 `AutoGPTQConfig` 的实现（`auto_gptq.py:219`）：

```python
@classmethod
def override_quantization_method(cls, hf_quant_cfg, user_quant, hf_config=None):
    quant_method = hf_quant_cfg.get("quant_method", "").lower()
    if quant_method != "gptq":           # 我只认 gptq checkpoint
        return None
    is_valid_user_quant = user_quant is None or user_quant in (
        "gptq", "gptq_marlin", "auto_gptq", "marlin")
    if is_valid_user_quant:
        return cls.get_name()             # 返回 "auto_gptq"，接管
    return None
```

**为什么这个设计聪明**：
- 它解耦了"checkpoint 声明的格式名"和"vLLM 内部真正用的实现名"。HF 生态用 `"gptq"`，vLLM 内部统一走 `AutoGPTQConfig`，再在内部决定调 Marlin 还是别的 kernel。
- `overrides` 列表的**顺序就是优先级**。GPTQ 系列里 `auto_gptq` 排最前，所以默认就用 Marlin 加速版。
- 这是一个**责任链（Chain of Responsibility）**模式：第一个声称能处理的就接管。

> 💡 对比：`override_quantization_method` 是"类方法"（不依赖实例，只看 checkpoint 和用户参数），而 `get_quant_method` 是"实例方法"（依赖已解析好的配置）。前者解决"用哪种量化"，后者解决"这一层用哪个 kernel"。

## 七、`get_quant_method`：一层一策略（步骤 ⑤）

`QuantizationConfig.get_quant_method(layer, prefix)` 是工厂之后的第二级分发。它根据 **layer 的类型**返回不同的 Method。看 `Fp8Config`（`fp8.py:175`）：

```python
def get_quant_method(self, layer, prefix) -> QuantizeMethodBase | None:
    if isinstance(layer, LinearBase):
        if is_layer_skipped(prefix, self.ignored_layers, ...):
            return UnquantizedLinearMethod()     # 被忽略的层，退回普通 Linear
        if not self.is_checkpoint_fp8_serialized: # 在线 FP8 量化
            return Fp8PerTensorOnlineLinearMethod()
        else:                                      # 离线 FP8 checkpoint
            return Fp8LinearMethod(self)
    elif isinstance(layer, RoutedExperts):        # MoE 专家层
        ...
        return Fp8MoEMethod(self, layer)
    elif isinstance(layer, Attention):            # 注意力层（KV cache 量化）
        return Fp8KVCacheMethod(self)
    return None
```

**这里暴露了 vLLM 的三类量化对象**：
1. **Linear 量化**（最常见）：`Fp8LinearMethod`
2. **MoE 量化**（专家网络 fused）：`Fp8MoEMethod`
3. **KV cache 量化**：`Fp8KVCacheMethod`

同一个 `Fp8Config` 能同时管理这三种，因为 FP8 可以应用在权重、专家、KV cache 三个不同位置。这种"一个 Config 类按 layer 类型分发多个 Method"的设计，让量化方法作者只写一份配置解析逻辑，复用给所有 layer 类型。

### 7.1 LinearBase：消费方

`vllm/model_executor/layers/linear.py:268`

```python
class LinearBase(nn.Module):
    def __init__(self, ..., quant_config=None, prefix=""):
        ...
        self.quant_method: QuantizeMethodBase
        if quant_config is None:
            self.quant_method = UnquantizedLinearMethod()
        elif quant_method := quant_config.get_quant_method(self, prefix=prefix):
            self.quant_method = quant_method
        else:
            raise ValueError("All linear layers should support quant method.")
```

注意 `prefix`（这一层在 state_dict 里的全名，如 `model.layers.0.self_attn.q_proj`）被传进去了——这是 `is_layer_skipped` 判断"这层要不要量化"的依据，也是 dynamic per-layer override（见 auto_gptq.py:80 `get_dynamic_override`）的依据。

## 八、CPA 三步走：以 FP8 Linear 为例（步骤 ⑥⑦⑧⑨）

`Fp8LinearMethod`（`fp8.py:266`）是入门最好的例子，因为它不涉及复杂的 pack/interleave（FP8 权重就是 `float8_e4m3fn` 的 weight + 一个 scale），能让你聚焦在**机制**而不是**位操作**上。

### 8.1 Create（`create_weights`，fp8.py:321）

```python
def create_weights(self, layer, input_size_per_partition,
                   output_partition_sizes, ..., **extra_weight_attrs):
    output_size_per_partition = sum(output_partition_sizes)
    layer.logical_widths = output_partition_sizes
    layer.input_size_per_partition = input_size_per_partition
    ...

    # ① 权重本体：FP8 dtype
    weight = create_fp8_weight_parameter(
        output_size_per_partition, input_size_per_partition, weight_loader)
    layer.register_parameter("weight", weight)

    # ② 权重 scale：per-tensor 或 block-wise 两种布局
    if not self.block_quant:
        scale = create_fp8_scale_parameter(PerTensorScaleParameter, ...)
        layer.register_parameter("weight_scale", scale)
    else:
        scale = create_fp8_scale_parameter(BlockQuantScaleParameter, ...)
        layer.register_parameter("weight_scale_inv", scale)  # DeepSeek 命名
```

关键观察：量化 Linear **不再是一个 `nn.Linear`**，它通过 `register_parameter` 手动注册了 `weight`（FP8）和 `weight_scale`（FP32）两个参数。这就是"替换权重布局"的实质。

### 8.2 Process（`process_weights_after_loading`）

权重从磁盘流式加载完后调用。FP8 在这里通常做：把 weight 转成 `float8_e4m3fn`、处理 TP 切分后的 scale、必要时 transpose 给 cutlass。后续讲 FP8 专讲会深入。

### 8.3 Apply（`apply`，fp8.py:445）

```python
def apply(self, layer, x, bias=None):
    # 动态量化激活 → 调 torch._scaled_mm（FP8 GEMM kernel）
    return self.fp8_linear.apply_weights(layer, x, bias)
```

底层最终落到 `torch._scaled_mm`（cutlass 实现）或 DeepGEMM 或 Marlin，这就是 kernel 分发的终点。

## 九、本讲小结：把地图刻进脑子

回答开头的四个工程问题：

| 问题 | 答案 | 源码位置 |
|------|------|----------|
| ① 识别 checkpoint 类型 | 读 HF `quantization_config.quant_method`，经 `override` 责任链确定 | model.py:970 |

| ③ 替换权重布局 | Method 的 `create_weights` 注册量化参数，`process_weights_after_loading` 变换 | fp8.py:321 |
| ④ 可扩展 | `@register_quantization_config` 装饰器注入注册表 | __init__.py:58 |

**一句话总结**：vLLM 的量化是一个 **"字符串名 → Config 工厂 → 按 layer 分发 Method → CPA 三步管理权重"** 的分层策略系统，每一层都解耦，每一层都有扩展点。
