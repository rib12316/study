# 特性 #12：结构化输出 —— 用 Grammar Bitmask 从源头保证输出合法

> 学习阶段：AI Infra 基础储备 / 解码约束（新领域）
> 对应源码：`vllm/v1/structured_output/__init__.py`（StructuredOutputManager + bitmask 填充）+ `vllm/v1/structured_output/backend_xgrammar.py`（grammar matcher）+ `vllm/v1/sample/sampler.py`（bitmask 应用到 logits）
> 本讲定位：LLM 应用要求输出 JSON / function call / 特定格式时，朴素采样可能生成非法格式（缺括号、错类型）。这一讲回答：**怎么让 LLM 在采样时就保证每一步都符合 grammar，而不是事后重试？** 这是 function call / agent / 结构化数据抽取的工程基石。
> 干中学原则：本讲你要**亲手实现一个 mini grammar constrained sampler**——定义简单 grammar（如"数字串"或"JSON 键值"）+ 计算 bitmask（哪些 token 允许）+ 应用到 logits + 接受 token 推进状态机。

## 一、为什么需要结构化输出？（背景，面试必答）

### 1.1 朴素采样的格式灾难

你用 LLM 生成 JSON：`{"name": "Alice", "age": 30}`。但模型可能输出：
- `{"name": "Alice", "age": 30`（缺 `}`）
- `{"name": "Alice", "age": "thirty"}`（age 应该是数字却给了字符串）
- `Here is the JSON: {...}`（前面带废话）

应用层解析就崩了。朴素解法：
- **重试**：生成→解析失败→重新生成。浪费算力，且可能一直失败。
- **后处理修复**：正则补括号、强转类型。脆弱，复杂格式（嵌套 JSON）修不动。

### 1.2 Grammar Constrained Generation：从源头约束

结构化输出的核心思想：**不是事后修，而是采样时就不让非法 token 被选**。

做法：把目标格式（JSON schema / EBNF 语法 / regex）编译成一个**有限状态机（FSM）/ grammar matcher**。每生成一个 token，matcher 告诉你"当前位置，词表里哪些 token 是合法的"，生成一个 bitmask。采样前，把非法 token 的 logit 设为 -inf——**采样永远选不到非法 token**。

> 💡 面试一句话答：**结构化输出把目标格式（JSON/grammar/regex）编译成状态机，每步生成一个 bitmask 标记词表里哪些 token 合法，采样前把非法 token 的 logit 置 -inf，从源头保证输出符合格式——不是事后过滤而是事前约束，零重试、零后处理。**

## 二、直觉版讲解（先建立直觉，再看源码）

> 这一节用一个超小词表的例子，把结构化输出的道理讲透。**不碰源码，只讲原理**。看懂这一节，后面的源码章节就顺了。

### 2.0 问题：LLM 会"乱说"

假设你让 LLM 回答一个判断题，你**只要**它输出 `是` 或 `否` 两个字之一。

**没有约束时**，模型可能输出：`是` / `是的` / `我认为是` / `Yes` / `嗯，应该是吧`。你后面的程序要解析"是/否"，遇到 `我认为是` 就崩了。

**朴素的补救**都是**事后补救**——已经生成了，再想办法修（重试 / 后处理正则），又贵又脆。

### 2.1 核心思想：事前约束，不让它乱说

结构化输出的思路完全不同：**在采样那一步，就不让非法的词被选出来**。

LLM 生成下一个词时，本质是：① 模型给词表里每个词打分（logits）② 按分数高低采样一个。

**约束的做法**：你告诉系统"现在只允许 `是` 或 `否`"。系统在采样前，把 `的`、`我` 等所有其他词的分数**强行改成负无穷**（-∞）：

```
原始分数:   是=5, 的=8, 我=9, 否=2
约束后:     是=5, 的=-∞, 我=-∞, 否=2
```

负无穷的词，无论原来分多高，**采样时概率都是 0**，永远选不到。所以模型只能从 `是`、`否` 里选——**从源头保证合法**。

> 这就是结构化输出的全部精髓：**用一张"允许名单"屏蔽掉所有非法词，让模型只能选合法的。事前约束，不是事后补救。**

### 2.2 用一个完整例子走一遍

模拟一个**超小词表**的 LLM，词表 = `{A, B, C, D, 1, 2, 3}`（7 个词），规则（grammar）= **只能输出 A 或 B**。

**状态机**（规则的形式化）：

```
[开始] ──→ 只允许 A 或 B ──→ [结束]
```

- 开始状态：允许的词 = `{A, B}`（其他全禁止）
- 选了 A 或 B 之后：结束

**采样过程演示**。假设模型当前给每个词的分数（logits）是：

| 词 | A | B | C | D | 1 | 2 | 3 |
|----|---|---|---|---|---|---|---|
| 原始分数 | 3 | 2 | **9** | 5 | 1 | 0 | 0 |

注意 **C 的分数最高（9）**，正常采样大概率选 C——但它违反规则！

**应用约束**（把非 `{A,B}` 的词分数置 -∞）：

| 词 | A | B | C | D | 1 | 2 | 3 |
|----|---|---|---|---|---|---|---|
| 约束后分数 | 3 | 2 | **-∞** | -∞ | -∞ | -∞ | -∞ |

现在 softmax 之后，只有 A、B 有非零概率，C/D/1/2/3 概率都是 0。模型**必然**选 A 或 B。

**关键效果**：即使模型最想输出 C（分数 9 最高），约束后它也只能乖乖输出 A。这就是"事前约束"。

### 2.3 再进阶：状态机怎么处理"序列"

上面的例子只有一步。真实场景（如 JSON、数字串）是**多步**的，规则会变化。

例子：**只允许输出"数字串"**（如 `123`）。

状态机：
```
状态0(开始): 只允许数字 {1,2,3}
    ↓ 选了 1/2/3
状态1(数字中): 允许数字 {1,2,3} 或 结束符
    ↓ 选了结束符
状态2(结束)
```

每一步，"允许名单"取决于**当前在哪个状态**：

- 第 1 步（状态0）：允许 `{1,2,3}`，屏蔽 A/B/C/D
- 假设选了 `1` → 进入状态1
- 第 2 步（状态1）：允许 `{1,2,3,结束}`，屏蔽 A/B/C/D
- 假设选了 `2` → 还在状态1
- 第 3 步（状态1）：假设选了 `结束` → 进入状态2，生成完毕

最终输出 `12`，**每一步都合法**，不会冒出字母。

### 2.4 对应到实践脚本（practice_structured_output.py）

这份脚本就是让你**亲手实现**上面的过程：

- **实践 1**：数字串状态机 = 第 2.3 节那个。实现 `allowed_next_chars(state)` 返回当前状态允许哪些字符。
- **实践 2**（最核心）：受约束采样 = 第 2.2 节的"屏蔽"过程。关键一行：
  ```python
  masked = [logits[i] if vocab[i] in allowed else float("-inf")
            for i in range(len(vocab))]
  ```
  `vocab[i] in allowed` 就保留分数，否则置 `-∞`——**这一行实现"事前约束"**。
- **实践 3**：把规则换成 `{"字母键": 数字}`，状态更多，但原理一样。

验证里有个直观演示：词表里给字母 `a,b,c` 打分 10（超高）、数字 `0-9` 打分 -10（超低）。正常采样选字母；加数字约束后，**字母全被屏蔽，只能采样出数字**——即使它原本分数低。这就是结构化输出的威力。

### 2.5 一句话和一张图记住

**一句话**：结构化输出 = 在采样前用"允许名单"把非法词的分数置 -∞，让模型只能选合法的——事前约束，不是事后补救。

```
模型 logits:   A=3, B=2, C=9(最高!), D=5
                  │
                  ▼ 应用 grammar 约束（只允许 A/B）
              A=3, B=2, C=-∞, D=-∞
                  │
                  ▼ softmax + 采样
              只能选 A 或 B（C 即使原本最高也被屏蔽）
```

### 2.6 看完本节，回看源码就顺了

理解上面后，后面的源码章节就好懂了：

- **bitmask**（那张"允许名单"）：就是一个 0/1 数组，1=允许，0=禁止。`fill_bitmask` 就是根据 grammar 状态机生成这张名单。
- **`masked_fill_(~mask, -inf)`**：就是把名单里 0 的位置分数置 -∞，就是第 2.4 节那行代码。
- **`accept_tokens`**：模型选了一个合法词后，状态机推进到下一状态，下一状态的允许名单可能不同（如第 2.3 节的状态0→状态1）。
- **grammar backend（xgrammar 等）**：帮你把 JSON schema / 正则 编译成状态机（你不用手写状态机，库帮你做）。

## 三、核心机制：Bitmask 采样（structured_output/__init__.py）

### 3.1 数据流

```
1. 请求带 grammar（JSON schema / EBNF）→ compile_grammar 编译成 matcher
2. 每生成一步前：
   - fill_bitmask(bitmask, index): matcher.fill_next_token_bitmask(bitmask, idx)
     生成 bitmask[index] = [vocab] 的 0/1 数组（1=允许，0=禁止）
3. 采样前：
   - logits.masked_fill_(~bitmask, -inf): 非法 token logit → -inf
4. 正常采样（greedy/topk/temperature）→ 必然选到合法 token
5. accept_tokens(tokens): matcher 接受 token，推进状态机到下一状态
6. 循环 2-5 直到 matcher.is_terminated()（grammar 完成）
```

### 3.2 bitmask 的形状

`backend_xgrammar.py:128 allocate_token_bitmask`：`[max_num_seqs, ceil(vocab/32)]`——用 bit 打包（每 int32 装 32 个 token 的允许/禁止），省内存。`fill_bitmask` 填充这个打包数组。

`grammar_bitmask` 方法（structured_output/__init__.py）收集 batch 里所有 structured 请求的 bitmask，返回 `[num_reqs, vocab_packed]`。

### 3.3 应用到 logits

sampler 拿到 bitmask 后（`sampler.py`）：
```python
# 解包 bitmask 成 bool[vocab]，对每个请求
# 非法位置 logit = -inf
logits[req_idx].masked_fill_(~allowed_mask, float("-inf"))
```
之后 greedy/topp/topk 采样都基于这个被 mask 过的 logits，**自然只选合法 token**。

## 四、Grammar Matcher：状态机推进（backend_xgrammar.py:136）

`XgrammarGrammar`（136行）封装了一个 grammar matcher，核心方法：

- `fill_bitmask(bitmask, idx)`（195行）：调 `matcher.fill_next_token_bitmask`，基于当前状态算合法 token
- `accept_tokens(tokens)`（152行）：接受生成的 token，推进 matcher 状态
- `rollback(n)`（190行）：回滚 n 个 token（用于 spec decode 被拒时）
- `is_terminated()`（198行）：grammar 是否完成（如 JSON 闭合）

**状态机的本质**：grammar 编译成一个 FSM。例如 JSON `{"key": value}` 的部分 FSM：
```
状态0: 期望 '{'      → 允许 token: '{'
状态1: 期望 '"'      → 允许: '"'（键名开始）
状态2: 键名字符      → 允许: 字母数字
状态3: 期望 '"'      → 允许: '"'（键名结束）
状态4: 期望 ':'      → 允许: ':'
状态5: 期望值        → 允许: 数字/字符串/true/false/null
...
```
每步 `fill_bitmask` 根据当前状态返回允许集，`accept_tokens` 推进到下一状态。

## 五、Token 级 vs 字符级：核心难点

这是结构化输出最tricky的地方，面试常追问。

Grammar 是**字符级**的（JSON 的 `{` 是一个字符），但 LLM 采样是**token 级**的（BPE token 可能是 `{"` 或 `name":` 这种多字符）。

问题：状态机期望下一个字符是 `{`，但 token `{"name":` 包含 `{` 又包含后面字符——这个 token 合法吗？要看后面字符是否也符合 grammar 的后续状态。

**xgrammar 的解法**：matcher 不只检查首字符，而是**模拟整个 token 的字符序列**能否被 grammar 接受（tokenize 感知）。`fill_next_token_bitmask` 遍历词表，对每个 token 检查"从这个状态出发，这个 token 的字符序列是否合法"，合法则在 bitmask 标 1。

这是为什么结构化输出有开销——每步要检查整个词表（vLLM 用 xgrammar 的 C++ 实现加速，并支持 `fill_bitmask_parallel_threshold` 并行）。

> 💡 你的实践用简化的"token=字符"模型（避免 BPE 复杂性），但讲义点明真实系统的 token-level 难点，让你知道为什么需要 xgrammar 这种专门库。

## 九、干中学实践任务（核心！）

> 在 `practice_structured_output.py` 里实现一个 mini grammar constrained sampler。
> 依赖：仅标准库（`math`）。不需要装 vllm/torch/xgrammar。
> 设计哲学：为避免 BPE 复杂性，用 **token=字符** 的简化模型（词表就是 ASCII 字符）。实现 grammar FSM + bitmask + 受约束采样，这是结构化输出的算法核心。

### 实践 1：数字串 grammar FSM（热身）
实现一个能匹配"纯数字串"（如 `12345`）的简单 FSM：
- `NumberGrammar` 类：状态 `START → DIGIT* → END`
- `allowed_next_chars(state) -> set[char]`：返回当前状态允许的字符集
  - START: 允许数字 0-9
  - DIGIT: 允许数字 0-9 或 END 触发符（如 `\n`）
- `accept(char) -> new_state`：接受一个字符，推进状态

验证：从 START，allowed 是数字；accept('1') 后还在 DIGIT，allowed 仍含数字。

### 实践 2：Bitmask + 受约束采样（核心）
实现 `constrained_sample(logits: list[float], grammar, state, temperature) -> (char, new_state)`：
- `mask = grammar.allowed_next_chars(state)`：得到允许字符集
- 把 logits 里**非允许字符**的位置设为 -inf
- 对剩余 logits 做 softmax + 采样（temperature 控制），返回采样字符
- `grammar.accept(char)` 推进状态

验证：构造 logits 让非数字字符概率高，但约束后只能采样出数字（即使 logits 偏向字母）。

### 实践 3：完整 JSON 键值 grammar（进阶）
实现一个能匹配 `{"<字母键>": <数字>}` 的 FSM（简化版 JSON）：
- 状态序列：`{` → `"` → 键名字母 → `"` → `:` → 数字 → `}`
- 实现完整状态转移 + allowed 集
- 跑一个完整生成循环：从空开始，每步 constrained_sample，直到 is_terminated

验证：无论 logits 如何偏向非法字符，生成的字符串一定是合法的 `{"key": 123}` 形式。

> 💡 实践 2 是灵魂。要点：① mask 是"白名单"（允许的设 1，禁止的设 0）② 应用是 `logits[~allowed] = -inf`（禁止的置负无穷）③ softmax 后禁止项概率严格为 0 ④ 采样必然落子允许集。这个"事前屏蔽"是结构化输出区别于"事后过滤"的根本。
